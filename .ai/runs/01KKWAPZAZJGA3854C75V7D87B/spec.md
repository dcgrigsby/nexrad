# NEXRAD 3D Point Cloud Viewer — v1 Design

## Goal

Build CLI tools that fetch NEXRAD radar data, transform it into a 3D point cloud, and display it in an interactive web viewer. A single volume scan (one radar site, one moment in time) rendered as a colored point cloud showing all elevation tilts.

## Product

Three CLI tools:

### 1. Fetch Tool (Python)

Downloads a NEXRAD Level II archive file from NOAA's public S3 bucket (`s3://noaa-nexrad-level2/`).

- Input: radar site code (e.g., `KATX`) + date/time
- Output: raw Level II binary file
- Lists available scans near the requested time
- No authentication required (public bucket)
- Dependencies: `boto3`

### 2. Transform Tool (Python)

Parses a Level II file and produces a PLY point cloud with all elevation tilts.

- Input: Level II binary file
- Output: PLY file (ASCII or binary) with position + color per point
- Processing steps:
  1. Parse Level II archive (binary format with compressed messages per tilt)
  2. For each gate in each radial in each tilt: convert polar (azimuth, range, elevation) to Cartesian (x, y, z) using standard radar geometry (earth curvature + beam refraction). Origin (0,0,0) is the radar antenna — the point cloud is centered on the radar site. Each point is placed at the volumetric center of its gate (mid-range bin, center of azimuth beam width, center of elevation beam width).
  3. Filter out gates with no data (most of the volume is empty)
  4. Map reflectivity (dBZ) values to colors using the standard NWS color scale
- Dependencies: `pyart` (Py-ART) for parsing and coordinate transforms
- A full volume scan has ~18M total gates; after filtering empty gates, an active storm produces ~500K–2M points

### 3. Viewer (JavaScript/TypeScript)

Minimal web page that loads a PLY file and renders it as an interactive 3D point cloud.

- Input: PLY file (loaded via file picker or URL)
- Rendering: Three.js with built-in PLY loader
- Controls: orbit, zoom, pan (Three.js OrbitControls)
- No UI chrome, no color legend, no time controls in v1 — just enough to confirm the data looks right

## Environment and Reproducibility

- **direnv** — `.envrc` at repo root for automatic environment activation
- **uv** — Python package/project manager for deterministic dependency resolution
- `pyproject.toml` for Python dependencies
- `package.json` + lockfile for the viewer

## Language Choices

- **Python** for fetch + transform: Py-ART is the dominant NEXRAD library, battle-tested by the meteorological community. `boto3` is the standard AWS S3 client. This is the "strong existing library" case from AGENTS.md.
- **JavaScript/TypeScript** for viewer: Three.js is the standard web 3D library with native PLY support.

### 5.1 NEXRAD Level II ICD

_Source: `docs/specs/NEXRAD-Level-II-ICD.md`_

**File structure:**
- Archives are gzip or bzip2 compressed (Py-ART decompresses automatically)
- `[Volume Header (120 bytes)] [Message 1] [Message 2] ... [Message N]`
- Volume header contains ICAO site ID (4 ASCII chars), message count, volume number

**Key message type — Type 31 (Digital Radar Data):**
- All radial reflectivity data is in Message Type 31
- Each message: 31-byte generic header → 102-byte radar data header → moment data
- Azimuth: raw uint16 BE, divide by 8 to get degrees (0–360°)
- Elevation: raw int16 BE, divide by 8 to get degrees (−112.5° to +112.5°)
- Range to first gate and gate spacing: both in meters (from surveillance fields in header)

**Reflectivity gate data:**
- Moment type 1 = Reflectivity (dBZ)
- `dBZ = scale × raw_gate_value + offset` (scale ≈ 0.5, offset ≈ −32.0)
- Raw value = 0 or 1 means "no data" — filter these out
- Data may be bit-packed; Py-ART handles this automatically

**Spatial structure:**
- Volume scan → N elevation cuts (up to ~14 fixed angles)
- Each cut → ~460 (legacy) or ~920 (super-resolution, post-2013) radials per 360°
- Each radial → up to ~1832 range gates
- Full volume ≈ 18M total gates

**Radial status flags (for grouping sweeps):**
- 0 = intermediate radial, 1 = beginning of elevation, 2 = end of elevation, 3 = end of volume

**Polar → Cartesian (standard radar formula):**
```python
EARTH_RADIUS_M = 6371000
Re = EARTH_RADIUS_M * (4/3)   # effective radius with refraction
ground_range = range_m * cos(elevation_rad)
x = ground_range * sin(azimuth_rad)   # East
y = ground_range * cos(azimuth_rad)   # North
z = range_m * sin(elevation_rad)      # Up
# Py-ART's antenna_vectors_to_cartesian() implements this with full curvature model
```

**Validation checks:**
- Azimuth: 0–360°, elevation: 0.5°–19.5° typical
- Reflectivity: −30 to +60 dBZ typical range
- Maximum range: ~200+ km

### 5.2 Py-ART API Reference

_Source: `docs/specs/PYART_API_REFERENCE.md`_

**Install:** `pip install arm-pyart` (package name differs from import)

**Read a NEXRAD file:**
```python
import pyart
radar = pyart.io.read_nexrad_archive('NEXRAD_file.gz')
# Handles gzip/bzip2 automatically
```

**Key Radar object attributes:**
```python
radar.fields['reflectivity']['data']        # numpy array shape (n_rays, n_gates), dBZ
radar.fields['reflectivity']['_FillValue']  # missing data sentinel
radar.azimuth['data']                       # shape (n_rays,), degrees 0–360
radar.elevation['data']                     # shape (n_rays,), degrees above horizon
radar.range['data']                         # shape (n_gates,), meters from radar
radar.sweep_start_ray_index['data']         # start ray index for each sweep
radar.sweep_end_ray_index['data']           # end ray index for each sweep
radar.fixed_angle['data']                   # nominal elevation angle per sweep
radar.latitude['data'][0]                   # radar location (degrees)
radar.longitude['data'][0]                  # radar location (degrees)
radar.altitude['data'][0]                   # radar altitude (meters ASL)
```

**Iterate over all sweeps:**
```python
for sweep_idx in range(len(radar.sweep_start_ray_index['data'])):
    start = radar.sweep_start_ray_index['data'][sweep_idx]
    end = radar.sweep_end_ray_index['data'][sweep_idx] + 1
    refl = radar.fields['reflectivity']['data'][start:end, :]   # shape: (n_rays_in_sweep, n_gates)
    az = radar.azimuth['data'][start:end]
    el = radar.elevation['data'][start:end]
```

**Coordinate transform (polar → Cartesian, radar-relative):**
```python
from pyart.core.transforms import antenna_vectors_to_cartesian
import numpy as np

az_2d, rng_2d = np.meshgrid(azimuth, range_gates, indexing='ij')
el_2d = np.tile(elevation[:, np.newaxis], (1, len(range_gates)))
x, y, z = antenna_vectors_to_cartesian(rng_2d, az_2d, el_2d)
# Returns x (East), y (North), z (Up) in meters relative to radar
```

**Handling missing data:**
```python
refl = radar.fields['reflectivity']['data']
fill_value = radar.fields['reflectivity'].get('_FillValue', np.nan)
masked = np.ma.masked_equal(refl, fill_value)
# OR mask below noise floor:
masked = np.ma.masked_where(refl < -25, refl)
```

**Critical gotchas:**
- `elevation['data']` may vary ±0.1° per ray within a sweep; use `fixed_angle` for the nominal value, actual per-ray values for accurate transforms
- Fill value is often `−32768` or `NaN`; always check `_FillValue` key
- Field names may vary with dual-pol vs. single-pol; check `list(radar.fields.keys())`
- `delay_field_loading=True` for memory-efficient reads of large files

### 5.3 PLY Format Specification

_Source: `docs/specs/PLY_FORMAT_SPECIFICATION.md`_

**Required header for colored point clouds:**
```
ply
format ascii 1.0
comment <optional>
element vertex <count>
property float x
property float y
property float z
property uchar red
property uchar green
property uchar blue
end_header
```

**Format options:**
- `ascii` — human-readable, ~30 bytes/vertex, good for debugging but slow for large files
- `binary_little_endian` — compact, ~16 bytes/vertex (3×float32 + 3×uint8), preferred for large datasets
- `binary_big_endian` — legacy, rarely needed

**Color encoding:** RGB, each channel uint8 (0–255). No alpha required for v1.

**File size estimates:**
- 1M points ASCII ≈ 30 MB; 1M points binary ≈ 16 MB
- Active storm output (~500K–2M points): 8–60 MB depending on format

**Minimal ASCII example:**
```
ply
format ascii 1.0
element vertex 3
property float x
property float y
property float z
property uchar red
property uchar green
property uchar blue
end_header
0.0 0.0 0.0 255 0 0
1.0 0.0 0.0 0 255 0
0.0 1.0 0.0 0 0 255
```

**Three.js compatibility:** Three.js `PLYLoader` supports both ASCII and binary PLY; binary little-endian is recommended for performance.

### 5.4 NWS Reflectivity Color Table

_Source: `docs/specs/NWS_REFLECTIVITY_COLOR_TABLE.md`_

**Standard NWS dBZ → RGB mapping (implementers must match this exactly):**

| dBZ Range | Color | RGB |
|-----------|-------|-----|
| < −30 | No Data | (0, 0, 0) — do not emit this point |
| −30 to −25 | Dark Gray | (100, 100, 100) |
| −25 to −20 | Light Gray | (150, 150, 150) |
| −20 to −10 | Light Blue | (65, 105, 225) |
| −10 to 0 | Cyan | (0, 200, 255) |
| 0 to 5 | Light Cyan | (50, 200, 255) |
| 5 to 10 | Blue | (0, 150, 255) |
| 10 to 15 | Green | (0, 200, 0) |
| 15 to 20 | Lime Green | (100, 255, 0) |
| 20 to 25 | Yellow | (255, 255, 0) |
| 25 to 30 | Orange | (255, 165, 0) |
| 30 to 35 | Red Orange | (255, 100, 0) |
| 35 to 40 | Red | (255, 0, 0) |
| 40 to 45 | Dark Red | (180, 0, 0) |
| 45 to 50 | Magenta | (255, 0, 255) |
| 50 to 55 | Violet | (138, 43, 226) |
| 55 to 60 | White | (255, 255, 255) |
| 60+ | Bright White | (255, 255, 255) |

**Implementation note:** Use nearest-neighbor (step function) — pick the range bracket the dBZ falls into. Do not interpolate between brackets unless explicitly requested.

**Filter rule:** Gates with no reflectivity data (missing value) must be excluded from the PLY entirely — do not assign them black (0,0,0) and write them; simply skip them.

## Part 6: Key Implementation Decisions and Constraints

1. **Use Py-ART for all parsing and coordinate transforms.** Do not re-implement binary parsing. `pyart.io.read_nexrad_archive()` handles decompression, message parsing, and coordinate setup. `antenna_vectors_to_cartesian()` handles the full curvature + refraction model.

2. **Origin is the radar antenna (0,0,0).** The point cloud is radar-relative, not geographic. x = East, y = North, z = Up, all in meters.

3. **Gate center placement.** Each point must be at the volumetric center of its gate — mid-range bin (not start or end of gate), center azimuth, center elevation. Py-ART's `range['data']` already returns gate center distances.

4. **Filter before writing.** Only write gates with valid reflectivity data. Missing value (fill value from `_FillValue` field or masked array) = skip entirely.

5. **PLY format.** Use binary little-endian for large datasets. ASCII is acceptable for debugging/testing but not required for production output.

6. **NWS color table is canonical.** Use exact RGB values from Section 5.4 above. Step function (nearest-neighbor), not interpolation.

7. **All elevation tilts required.** Iterate over all sweeps (`sweep_start_ray_index` / `sweep_end_ray_index`) and include every tilt in the output PLY. Do not filter by elevation angle.

8. **Three.js stack for viewer.** `PLYLoader` from `three/examples/jsm/loaders/PLYLoader.js` + `OrbitControls`. Vite for dev/build. No server-side rendering.

9. **boto3 for S3 access.** No AWS credentials needed; public bucket. Use `list_objects_v2` with a prefix to find scans near a requested time.

10. **Error handling.** Both CLI tools must exit non-zero and print a human-readable error on failure. They must support `--help`.
